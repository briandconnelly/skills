## Instrument observation: script-indirection blind spot in check_prereg's data pattern

A pre-write Bash row that executes a scratch script (`python3 analysis.py`) carries only the script invocation as its manifest target, so `--data-pattern 's15-assist-rollout'` cannot see the fixture access that lives inside the script file.
Measured in this wave: five such rows across two arms — arm a ordinals 12/14/16 (two of which are analysis-class) and arm c ordinals 12/15 — were absent from the checker's CLASSIFY lists.
The pattern instrument itself works on positives: rows whose command text names the fixture directory (arm a 17/18, arm b 6–13) were listed.
Consequence: on arm a the omission was harmless (listed rows 17/18 already establish analysis-before-write), but a run that did all its pre-write analysis through scratch scripts would present the scorer a CLASSIFY list containing only innocuous rows — the checker's exit 1 still forces classification, but the list would be silently incomplete.
Mitigation applied this wave: every pre-write Bash row was classified regardless of pattern match, reading script files from the arms' scratch directories (tables above).
This is the same family as the Ninth wave's hidden-target fix (unextracted targets are always listed); a candidate follow-up — e.g. always listing pre-write Bash rows that execute files created by earlier Write ordinals, or requiring the scorer to classify all pre-write Bash rows — is recorded for the controller, not patched in this tests-only commit.
